title: "Developing a New Hardware Backend"
description: "How to retarget XLA to new CPU architectures and non-CPU accelerators using LLVM backends or from-scratch implementations."
---

XLA provides an abstract interface that a new architecture or accelerator can implement to create a backend for running ML programs. Retargeting XLA is significantly simpler and more scalable than implementing every existing op from a frontend framework such as PyTorch or TensorFlow for new hardware.

<Note>
This guide assumes knowledge of [LLVM](https://llvm.org), [Bazel](https://bazel.build/), and XLA internals. It is not a step-by-step tutorial.
</Note>

Most implementations fall into one of three scenarios:

<CardGroup cols={3}>
  <Card title="Scenario 1" icon="cpu">
    Existing CPU architecture not yet officially supported by XLA, with or without an existing LLVM backend.
  </Card>
  <Card title="Scenario 2" icon="microchip">
    Non-CPU-like hardware with an existing LLVM backend.
  </Card>
  <Card title="Scenario 3" icon="bolt">
    Non-CPU-like hardware without an existing LLVM backend.
  </Card>
</CardGroup>

<Note>
An LLVM backend can mean either one of the officially released LLVM backends, or a custom LLVM backend developed in-house.
</Note>

---

## Scenario 1: Existing CPU architecture not yet officially supported by XLA

XLA makes it easy to target different CPUs by using LLVM, because the main difference between XLA backends for CPUs is the code generated by LLVM. Start by looking at the existing [XLA CPU backend](https://github.com/openxla/xla/tree/main/xla/service/cpu).

**If the hardware vendor has an LLVM backend for their hardware**, it is straightforward to link that backend with the LLVM built alongside XLA:

- In **JIT mode**, the XLA CPU backend emits code for the host CPU automatically.
- For **ahead-of-time (AOT) compilation**, use [`xla::AotCompilationOptions`](https://github.com/openxla/xla/tree/main/xla/service/compiler.h) to provide an LLVM triple that configures the target architecture.

**If there is no existing LLVM backend** but another kind of code generator exists, it is still possible to reuse most of the existing CPU backend logic.

```c++
// Configure AOT compilation for a custom CPU target via LLVM triple
xla::AotCompilationOptions options(target_triple);
// Pass options to xla::Compiler::RunBackend(...)
```

## Scenario 2: Non-CPU-like hardware with an existing LLVM backend

Model a new [`xla::Compiler`](https://github.com/openxla/xla/tree/main/xla/service/compiler.h) implementation on the existing [`xla::CPUCompiler`](https://github.com/openxla/xla/tree/main/xla/service/cpu/cpu_compiler.cc) and [`xla::GPUCompiler`](https://github.com/openxla/xla/tree/main/xla/service/gpu/nvptx_compiler.cc) classes, since these already emit LLVM IR.

Depending on the nature of the hardware, many aspects of LLVM IR generation may need to change, but a substantial amount of code can be shared with the existing backends.

<Tip>
The [GPU backend](https://github.com/openxla/xla/tree/main/xla/service/gpu/) is the recommended model for non-CPU-like ISAs. Other kinds of hardware — for example, DSPs such as Hexagon (which has an upstream LLVM backend) — can reuse parts of the LLVM IR emission logic while replacing the parts that are architecture-specific.
</Tip>

**Key classes to implement or subclass:**

| Class | Header | Purpose |
|---|---|---|
| `xla::Compiler` | `xla/service/compiler.h` | Encapsulates compilation of HLO into an `xla::Executable` |
| `xla::CPUCompiler` | `xla/service/cpu/cpu_compiler.cc` | Reference CPU compiler using LLVM IR |
| `xla::GPUCompiler` | `xla/service/gpu/nvptx_compiler.cc` | Reference GPU compiler (non-CPU-like ISA) |

## Scenario 3: Non-CPU-like hardware without an existing LLVM backend

If LLVM cannot be used at all, a new backend for XLA must be implemented from scratch. This option requires the most effort.

<Steps>
  <Step title="Implement StreamExecutor">
    [`StreamExecutor`](https://github.com/openxla/xla/tree/main/xla/stream_executor/stream_executor.h) manages device interaction. Not all methods are required for every device — see existing `StreamExecutor` implementations for which methods are necessary.
  </Step>
  <Step title="Implement xla::Compiler">
    [`xla::Compiler`](https://github.com/openxla/xla/tree/main/xla/service/compiler.h) encapsulates the compilation of an HLO computation into an `xla::Executable`. This is the main entry point for your backend's compilation pipeline.
  </Step>
  <Step title="Implement xla::Executable">
    [`xla::Executable`](https://github.com/openxla/xla/tree/main/xla/service/executable.h) is used to launch a compiled computation on the platform. It holds the compiled artifact and provides the `ExecuteOnStream` interface.
  </Step>
  <Step title="Implement xla::TransferManager">
    [`xla::TransferManager`](https://github.com/openxla/xla/tree/main/xla/service/transfer_manager.h) enables backends to provide platform-specific mechanisms for constructing XLA literal data from device memory handles. It encapsulates host-to-device and device-to-host data transfers.
  </Step>
</Steps>

**Summary of required classes:**

<AccordionGroup>
  <Accordion title="StreamExecutor">
    Provides the platform abstraction: device memory allocation, kernel launch, stream management, and more. Implement only the subset of methods that your hardware supports.

    Header: `xla/stream_executor/stream_executor.h`
  </Accordion>
  <Accordion title="xla::Compiler">
    Takes an HLO module and a `StreamExecutor` and produces an `xla::Executable`. Subclass this and implement `RunBackend` and `RunHloPasses`.

    Header: `xla/service/compiler.h`
  </Accordion>
  <Accordion title="xla::Executable">
    Holds the compiled binary and provides execution via `ExecuteOnStream`. Also manages profiling and metadata.

    Header: `xla/service/executable.h`
  </Accordion>
  <Accordion title="xla::TransferManager">
    Handles the layout and transfer of `xla::Literal` values between host and device memory. The default implementation may be sufficient for some devices.

    Header: `xla/service/transfer_manager.h`
  </Accordion>
</AccordionGroup>

## Choosing the right approach

<Tabs>
  <Tab title="CPU with LLVM">
    Link the LLVM target backend into XLA's build. Use `xla::AotCompilationOptions` to supply the LLVM triple for cross-compilation. Minimal new code required.
  </Tab>
  <Tab title="Non-CPU with LLVM">
    Subclass `xla::Compiler`. Reuse LLVM IR emission from `xla::CPUCompiler` or `xla::GPUCompiler` and override the parts specific to your ISA.
  </Tab>
  <Tab title="Non-CPU without LLVM">
    Implement `StreamExecutor`, `xla::Compiler`, `xla::Executable`, and `xla::TransferManager` from scratch, using the GPU backend as a structural reference.
  </Tab>
</Tabs>
