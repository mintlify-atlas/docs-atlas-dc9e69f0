There are several ways to write unit tests for HLO passes. This page describes the
preferred approaches for consistency and readability.

<Note>
The preferred method is to use FileCheck with `CHECK` lines interleaved directly in
the HLO module text. This is more readable than writing separate input and expected
output modules.
</Note>

## FileCheck with interleaved CHECK lines

Most HLO passes can be tested using
[`FileCheck`](https://llvm.org/docs/CommandGuide/FileCheck.html) with `CHECK` lines
interleaved in the input HLO module text.

<Warning>
Always use `// CHECK` (C++ style) instead of `; CHECK` (assembly/HLO comment style)
as the FileCheck delimiter. This matches the MLIR-based testing convention.
</Warning>

### Example: priority fusion test

```cpp
TEST_F(PriorityFusionTest, FuseBroadcastIntoBitcastConsumers) {
  absl::string_view kHlo = R"(
    HloModule test_module

    // CHECK: ENTRY main
    ENTRY main {
      // CHECK-NEXT: %[[PARAM:.*]] = f32[96]{0} parameter(0)
      param_0 = f32[96]{0} parameter(0)
      broadcast = f32[8,96,128,7]{3,2,1,0} broadcast(param_0), dimensions={1}
      bitcast.6079.2 = f32[8,24,4,128,7]{4,3,2,1,0} bitcast(broadcast)
      // CHECK-NEXT: ROOT %{{.*}} fusion(%[[PARAM]]) {{.*}}
      ROOT transpose.1990.2 = f32[8,24,128,7,4]{4,3,2,1,0} transpose(bitcast.6079.2),
                               dimensions={0,1,3,4,2}
    }
  )";
  RunAndFilecheckHloRewrite(kHlo, std::move(priority_fusion_));
}
```

The test runner `RunAndFilecheckHloRewrite` applies the pass to the HLO module and
verifies the output matches the `CHECK` patterns.

### Common CHECK directives

| Directive | Matches |
| --------- | ------- |
| `// CHECK: pattern` | Any line containing `pattern` |
| `// CHECK-NEXT: pattern` | The very next non-empty line |
| `// CHECK-NOT: pattern` | Asserts `pattern` does NOT appear |
| `// CHECK-SAME: pattern` | Same line as the previous CHECK |
| `// CHECK-DAG: pattern` | Any order in a block (useful for unordered results) |
| `// CHECK: %[[VAR:.*]] = ...` | Capture a name into `VAR` for later use with `%[[VAR]]` |

### Why interleaving is preferred

The codebase previously used separate input and expected modules. Interleaved `CHECK`
lines are now preferred because:

- Better readability — the expected output is co-located with the input it tests
- Same style as MLIR-based tests (e.g., [`stablehlo_aggressive_folder.mlir`](https://github.com/openxla/stablehlo/blob/main/stablehlo/tests/transforms/stablehlo_aggressive_folder.mlir#L31-L39))
- Easier to update when pass output changes

## LIT runner with hlo-opt

Where feasible, use the [`LIT`](https://llvm.org/docs/CommandGuide/lit.html) runner with
`hlo-opt` and place `CHECK` lines locally next to the input IR they correspond to.

### Example: GPU test with LIT

```hlo
// RUN: hlo-opt %s --platform=gpu --stage=llvm-before-optimizations \
// RUN:   --xla_gpu_target_config_filename=%S/../../../backends/gpu/specs/%{GPU}.txtpb \
// RUN:   | FileCheck --check-prefixes=CHECK-%{PTX} %s

HloModule Test, is_scheduled=true

fused_computation {
  param_0 = f32[100,200]{1,0} parameter(0)
  ROOT b.1 = f32[200,100]{1,0} transpose(f32[100,200]{1,0} param_0), dimensions={1,0}
}

ENTRY main {
  a = f32[100, 200]{1,0} parameter(0)
  // CHECK-PTX:         call void @llvm.nvvm.barrier0
  // CHECK-GCN:         call void @llvm.amdgcn.s.barrier
  ROOT wrapped_b = f32[200,100]{1,0} fusion(f32[100,200]{1,0} a),
    kind=kInput, calls=fused_computation
}
```

The `RUN:` line at the top invokes `hlo-opt` and pipes the output to `FileCheck`. The
`%{GPU}` and `%{PTX}` substitutions are filled in by the LIT configuration for the
target hardware.

## Automated CHECK generation

Writing `CHECK` lines manually can be tedious for complex passes. Use
[`generate_hlo_test_checks.py`](https://github.com/openxla/xla/tree/main/xla/hlo/tools/generate_hlo_test_checks.py)
to automatically insert generated `CHECK` directives above each test case in an HLO file.

```bash
python xla/hlo/tools/generate_hlo_test_checks.py \
  --pass=my-pass \
  input.hlo \
  --output=input_with_checks.hlo
```

<Warning>
The script assumes the pass's current behavior is correct. Always review generated
`CHECK` lines to confirm they match the output you actually expect before committing.
</Warning>

## Test infrastructure

### RunAndFilecheckHloRewrite

The `RunAndFilecheckHloRewrite` helper is used in C++ unit tests to apply a pass and
run FileCheck on the result:

```cpp
// Signature
void RunAndFilecheckHloRewrite(
    absl::string_view hlo,
    HloPassInterface&& pass,
    std::optional<absl::string_view> expected = std::nullopt,
    std::function<void(HloModule*)> after_pass_checks = nullptr);
```

When `expected` is not provided, the test uses FileCheck `CHECK` lines embedded in `hlo`.

### Test base classes

| Class | Use case |
| ----- | -------- |
| `HloHardwareIndependentTestBase` | Tests that don't require hardware |
| `HloPjRtInterpreterReferenceMixin<HloPjRtTestBase>` | Tests comparing against reference interpreter |

## What not to do: graph traversal

Do not write tests that traverse leaf nodes of the result graph and match against expected
operations. These tests are:

- Tedious to write
- Difficult to read quickly
- Hard to debug and fix when they fail

Use FileCheck or LIT instead.

```cpp
// BAD: graph traversal approach
EXPECT_EQ(root->opcode(), HloOpcode::kFusion);
EXPECT_EQ(root->operand(0)->opcode(), HloOpcode::kParameter);
// ...many more checks

// GOOD: FileCheck approach
absl::string_view kHlo = R"(
  ENTRY main {
    // CHECK-NEXT: ROOT %{{.*}} = f32[100]{0} fusion(...)
    ...
  }
)";
RunAndFilecheckHloRewrite(kHlo, std::move(my_pass_));
```

## Adding a new pass: end-to-end workflow

<Steps>
  <Step title="Write the pass">
    Implement your HLO pass in `xla/hlo/transforms/` (hardware-independent) or
    `xla/service/gpu/transforms/` (GPU-specific).
  </Step>
  <Step title="Register with hlo-opt">
    Add `RegisterPass<MyPass>(...)` to the appropriate registration file. See
    [XLA Tools - Registering a new pass](/reference/tools) for details.
  </Step>
  <Step title="Run the pass and capture output">

    ```bash
    hlo-opt --passes=my-pass input.hlo > output.hlo
    ```

  </Step>
  <Step title="Generate CHECK lines">

    ```bash
    python xla/hlo/tools/generate_hlo_test_checks.py --pass=my-pass input.hlo
    ```

    Review the generated `CHECK` lines carefully.
  </Step>
  <Step title="Write the C++ test">

    ```cpp
    TEST_F(MyPassTest, TransformsExpectedPattern) {
      absl::string_view kHlo = R"(
        HloModule test_module
        ENTRY main {
          // CHECK-NEXT: ...expected output...
          param_0 = f32[100]{0} parameter(0)
          ROOT result = f32[100]{0} my-op(param_0)
        }
      )";
      RunAndFilecheckHloRewrite(kHlo, MyPass());
    }
    ```

  </Step>
  <Step title="Run with Bazel">

    ```bash
    bazel test //xla/service/gpu/transforms:my_pass_test
    ```

  </Step>
</Steps>
