title: "Broadcasting"
description: "How XLA makes arrays with different shapes compatible for element-wise operations, including explicit broadcast dimensions and degenerate-dimension broadcasting."
---

Broadcasting is the process of making arrays with different shapes compatible so that element-wise operations can be applied across them. XLA's broadcasting is more explicit than NumPy's: you must always state _how_ dimensions correspond, and the compiler never infers broadcast dimensions on your behalf.

## Scalars are always broadcastable

Any scalar can be combined with any array without specifying broadcast dimensions. The scalar is applied independently to each element of the array.

```hlo
-- Adding scalar 7 to a 2×3 matrix
result = f32[2,3] add(matrix, scalar)
```

Conceptually:

```
|1 2 3| + 7 = |8  9  10|
|4 5 6|       |11 12 13|
```

## Broadcasting a lower-dimensional array onto a higher-dimensional array

When one operand has fewer dimensions than the other, you must supply a `broadcast_dimensions` tuple. This tuple maps each dimension of the **lower-dimensional** operand to a dimension of the **higher-dimensional** operand.

### Vector added to a matrix

```hlo
-- v has shape f32[3], M has shape f32[2,3]
-- broadcast_dimensions=(1) maps v's dim 0 → M's dim 1
result = f32[2,3] add(M, v), broadcast_dimensions={1}
```

With `broadcast_dimensions={1}`, the vector `[7, 8, 9]` is replicated along dimension 0 (rows):

```
|7 8 9| → |7 8 9|
           |7 8 9|

|1 2 3| + |7 8 9| = |8  10 12|
|4 5 6|   |7 8 9|   |11 13 15|
```

Alternatively, using `broadcast_dimensions={0}` maps the vector to rows, replicating along columns:

```
|7|   |7 7 7|
|8| → |8 8 8|
|9|   |9 9 9|
```

<Warning>
For a 2×3 matrix and a 3-element vector, `broadcast_dimensions={0}` is invalid because the vector length (3) does not match dimension 0 of the matrix (2). XLA will reject the program at compile time.
</Warning>

### Formal definition

For a higher-dimensional array of shape `M×N×P×Q` and a lower-dimensional vector of size `T`, the vector can be matched to any single dimension:

```
          MxNxPxQ

dim 3:          T
dim 2:        T
dim 1:      T
dim 0:    T
```

In each case `T` must equal the size of the matched dimension. For a 2D lower-dimensional operand of shape `T×V`, two broadcast dimensions are required:

```
          MxNxPxQ
dim 2,3:      T V
dim 1,2:    T V
dim 0,3:  T     V
```

The broadcast dimension tuple must be **strictly increasing** — it is illegal to swap the order of matched dimensions.

### Multi-dimensional example

Add a `1×2` matrix to a `4×3×1` array using `broadcast_dimensions={1, 2}`:

1. The `1×2` matrix broadcasts up to `4×1×2` (an intermediate 3D shape where M is determined by the larger operand).
2. The resulting `4×1×2` intermediate combines with the `4×3×1` array via degenerate-dimension broadcasting to produce `4×3×2`.

## Broadcasting with degenerate dimensions

Two arrays with the **same number of dimensions** can be broadcast together if every corresponding dimension pair is either equal or one of them is `1`. A dimension of size `1` is called *degenerate* — it is stretched to match the other operand.

The result shape takes the maximum size at each dimension index.

<CodeGroup>

```hlo compatible pairs
-- (2,1) and (2,3) -> (2,3)
-- (1,2,5) and (7,2,5) -> (7,2,5)
-- (7,2,5) and (7,1,5) -> (7,2,5)
```

```hlo incompatible pair
-- (7,2,5) and (7,2,6) -> ERROR: dimension 2 is 5 vs 6, neither is 1
```

</CodeGroup>

A special case: if each operand has a degenerate dimension at a _different_ position, the result is an outer operation:

```hlo
-- (2,1) and (1,3) -> (2,3)  (outer product shape)
result = f32[2,3] add(a, b)  -- a is f32[2,1], b is f32[1,3]
```

## Combining both forms of broadcasting

Lower-dimensional broadcasting and degenerate-dimension broadcasting can happen in a single operation.

**Example:** add a vector of size 4 (`f32[4]`) to a `1×2` matrix (`f32[1,2]`) using `broadcast_dimensions={0}`:

<Steps>
  <Step title="Lower-dimensional broadcast">
    The vector `[1, 2, 3, 4]` is broadcast up to a `4×M` matrix. `broadcast_dimensions={0}` maps the vector's dim 0 to the result's dim 0. `M` is chosen to match the corresponding dimension of the `1×2` matrix, giving a `4×2` intermediate:

    ```
    |1 1|
    |2 2|
    |3 3|
    |4 4|
    ```
  </Step>
  <Step title="Degenerate-dimension broadcast">
    The `4×2` intermediate is added to the `1×2` matrix. Dimension 0 of the matrix (size 1) is stretched to 4:

    ```
    |1 1|   |5 6|   |6  7|
    |2 2| + |5 6| = |7  8|
    |3 3|   |5 6|   |8  9|
    |4 4|   |5 6|   |9 10|
    ```
  </Step>
</Steps>

## XlaBuilder API

The `broadcast_dimensions` argument is accepted by binary ops in `XlaBuilder`. For example:

```cpp
// Add a vector v (shape [3]) to a matrix M (shape [2,3])
// broadcast_dimensions={1} maps v's dim 0 to M's dim 1
XlaOp result = Add(M, v, /*broadcast_dimensions=*/{1});
```

This form of broadcasting is referred to as "InDim" broadcasting in the XLA source code.

<Note>
XLA requires explicit broadcast dimensions whenever operand ranks differ (except for scalars). If you omit `broadcast_dimensions` on a non-scalar operand with a different rank, the operation is invalid and compilation will fail.
</Note>
