HLO indexing analysis is a [dataflow analysis](https://en.wikipedia.org/wiki/Data-flow_analysis) that describes how elements of one tensor relate to elements of another through *indexing maps*. For any HLO instruction, indexing analysis can answer: "which elements of operand `B` are needed to compute element `(i, j, k)` of output `A`?"

## Motivating example

Given a broadcast from `f32[20]` to `f32[10, 20, 30]`:

```hlo
p0 = f32[20] parameter(0)
bc0 = f32[10, 20, 30] broadcast(p0), dimensions={1}
```

The output-to-input indexing map is:

```
(i, j, k) -> (j)
domain: i in [0, 9], j in [0, 19], k in [0, 29]
```

To compute any element at position `(i, j, k)` of the output, XLA reads element `j` from the input — regardless of `i` or `k`, which are broadcast dimensions.

## Why indexing analysis exists

XLA uses bespoke solutions in several places to reason about tensor element access patterns. Indexing analysis provides a single reusable component for all of them.

<AccordionGroup>
  <Accordion title="Coalescing">
    Memory coalescing on GPUs requires that threads in a warp access contiguous memory locations. Knowing the full indexing map from output elements to input elements makes it possible to determine, for non-trivial access patterns (transposes, gathers, etc.), whether accesses will be coalesced.
  </Accordion>
  <Accordion title="Operand utilization">
    Operand utilization measures how much of each input is consumed when the output is fully used. Indexing maps make this computable for arbitrary ops, including those with strided or partial access patterns.
  </Accordion>
  <Accordion title="Tile propagation">
    A tile is a hyper-rectangular subset of a tensor described by offsets, sizes, and strides. Given a tile of the output, tile propagation computes the corresponding tile of each input. This feeds directly into GPU kernel code generation and fusion decisions. Indexing maps make tile propagation generic and robust across all HLO ops.
  </Accordion>
</AccordionGroup>

## Indexing maps

An *indexing map* combines:

1. A symbolically expressed function mapping every element of tensor `A` to ranges of elements in tensor `B`.
2. Constraints (the *domain*) on valid argument values.

### Variable categories

Function arguments are divided into three categories:

| Category | Notation | Description |
|----------|----------|-------------|
| Dimension variables | `d0, d1, ...` | Index coordinates of the tensor we are mapping from; statically known |
| Range variables | `s0, s1, ...` | Span a set of elements in the target tensor; needed for one-to-many mappings like reductions |
| Runtime variables | `rt0, rt1, ...` | Only known at execution time, e.g. dynamic offsets in `dynamic-slice` |

The general signature of an indexing function from `A` to `B` for operation `x` is:

```
map_ab(index in A)[range variables]{runtime variables} -> (index in B)
```

### Reduce example

For `f32[4, 8] out = reduce(f32[2, 4, 8, 16] in, 0), dimensions={0, 3}`:

<Tabs>
  <Tab title="Output → input">
    ```
    (d0, d1)[s0, s1] -> (s0, d0, d1, s1)
    domain:
    d0 in [0, 3],
    d1 in [0, 7],
    s0 in [0, 1],
    s1 in [0, 15]
    ```

    Range variables `s0` and `s1` cover the two reduced dimensions. A single output element `(d0, d1)` requires reading a 2×16 slice of the input.
  </Tab>
  <Tab title="Input → output">
    ```
    (d0, d1, d2, d3) -> (d1, d2)
    domain:
    d0 in [0, 1], d1 in [0, 3], d2 in [0, 7], d3 in [0, 15]
    ```

    Dimensions 0 and 3 (the reduced axes) are dropped from the output index.
  </Tab>
</Tabs>

<Note>
XLA indexing analysis primarily works *from outputs toward inputs* — the opposite direction from most dataflow analyses. When reasoning about "indexing of B" in a chain `C = op1(A, B); E = op2(C, D)`, this means "mapping of E's elements into B's elements".
</Note>

## Implementation: AffineMap + HLO semantics

Indexing analysis is built on [`mlir::AffineMap`](https://github.com/llvm/llvm-project/blob/main/mlir/include/mlir/IR/AffineMap.h) from the MLIR project and adds HLO-specific constraint tracking. A typical `AffineMap` looks like:

```
(d0)[s0, s1] -> (s0 + 5, d0 * 2, s1 * 3 + 50)
```

`AffineMap` has two parameter kinds — *dimensions* (`d`) and *symbols* (`s`) — but carries no constraint metadata. The `IndexingMap` struct wraps an `AffineMap` with explicit bounds:

```cpp
struct Interval {
  int64_t lower;
  int64_t upper;
};

class IndexingMap {
  mlir::AffineMap affine_map_;

  // Box constraints for dimension variables (usually match output shape)
  std::vector<Variable> dim_vars_;

  // Ranges for range variables (e.g. reduction dims, broadcast dims)
  std::vector<Variable> range_vars_;

  // Feasible intervals for runtime variables (e.g. dynamic-slice offsets)
  std::vector<Variable> rt_vars_;

  // Auxiliary constraints: <affine_expr> in <interval>
  llvm::DenseMap<mlir::AffineExpr, Interval> constraints_;
};
```

## Indexing maps for common ops

### Elementwise

Identity mapping in both directions:

```hlo
p0 = f32[10, 20] parameter(0)
p1 = f32[10, 20] parameter(1)
output = f32[10, 20] add(p0, p1)
```

```
output -> p0: (d0, d1) -> (d0, d1),  d0 in [0, 9], d1 in [0, 19]
p0 -> output: (d0, d1) -> (d0, d1),  d0 in [0, 9], d1 in [0, 19]
```

### Broadcast

Broadcast removes dimensions on the output→input path and introduces range variables on the input→output path:

```hlo
p0 = f32[20] parameter(0)
bc0 = f32[10, 20, 30] broadcast(p0), dimensions={1}
```

```
-- output -> input (drops broadcast dims)
(d0, d1, d2) -> (d1),
domain: d0 in [0, 9], d1 in [0, 19], d2 in [0, 29]

-- input -> output (adds range vars for broadcast dims)
(d0)[s0, s1] -> (s0, d0, s1),
domain: d0 in [0, 19], s0 in [0, 9], s1 in [0, 29]
```

Every element `d0` of the input maps to a `10×1×30` slice of the output, reflected by `s0` and `s1`.

### Transpose

The indexing map is a permutation of indices:

```hlo
p0 = f32[3, 12288, 6, 128] parameter(0)
transpose = f32[3, 6, 128, 12288] transpose(p0), dimensions={0, 2, 3, 1}
```

```
-- output -> input
(d0, d1, d2, d3) -> (d0, d3, d1, d2),
domain: d0 in [0, 2], d1 in [0, 5], d2 in [0, 127], d3 in [0, 12287]
```

### Slice

Strided access on output→input; the input→output map restricts to valid input elements via modular constraints:

```hlo
p0 = f32[10, 20, 50] parameter(0)
slice = f32[5, 3, 25] slice(p0), slice={[5:10:1], [3:20:7], [0:50:2]}
```

```
-- output -> input
(d0, d1, d2) -> (d0 + 5, d1 * 7 + 3, d2 * 2),
domain: d0 in [0, 4], d1 in [0, 2], d2 in [0, 24]

-- input -> output (restricted by mod constraints)
(d0, d1, d2) -> (d0 - 5, (d1 - 3) floordiv 7, d2 floordiv 2),
domain: d0 in [5, 9], d1 in [3, 17], d2 in [0, 48],
        (d1 - 3) mod 7 in [0, 0],
        d2 mod 2 in [0, 0]
```

### DynamicSlice

Runtime offset variables appear when slice indices are not statically known:

```hlo
src = s32[2, 2, 258] parameter(0)
of1 = s32[] parameter(1)
of2 = s32[] parameter(2)
of3 = s32[] parameter(3)
ds = s32[1, 2, 32] dynamic-slice(src, of1, of2, of3),
     dynamic_slice_sizes={1, 2, 32}
```

```
-- output -> src
(d0, d1, d2){rt0, rt1, rt2} -> (d0 + rt0, d1 + rt1, d2 + rt2),
domain:
d0 in [0, 0], d1 in [0, 1], d2 in [0, 31],
rt0 in [0, 1], rt1 in [0, 0], rt2 in [0, 226]
```

Runtime variable intervals are derived by assuming the entire slice stays in bounds.

## Indexing maps for fused ops

For a fusion cluster, the indexing map is the composition of individual op maps. This can expose surprising structure:

### Deduplicated maps

Two different paths through a fusion may produce the same indexing map:

```hlo
f {
  p0 = f32[20, 10, 50] parameter(0)
  lhs_transpose_1 = f32[10, 20, 50] transpose(p0), dimensions={1, 0, 2}
  lhs_e = f32[10, 20, 50] exponential(lhs_transpose_1)
  lhs_transpose_2 = f32[10, 50, 20] transpose(lhs_e), dimensions={0, 2, 1}
  rhs_transpose_1 = f32[50, 10, 20] transpose(p0), dimensions={2, 1, 0}
  rhs_log = f32[50, 10, 20] exponential(rhs_transpose_1)
  rhs_transpose_2 = f32[10, 50, 20] transpose(rhs_log), dimensions={1, 0, 2}
  ROOT output = f32[10, 50, 20] add(lhs_transpose_2, rhs_transpose_2)
}
```

Despite two separate chains of transposes, the composed output-to-`p0` map simplifies to a single expression:

```
(d0, d1, d2) -> (d2, d0, d1)
```

### Multiple maps for one input

When an input is used in different ways within the same fusion, multiple maps are produced:

```hlo
f {
  p0 = f32[1000, 1000] parameter(0)
  transpose_p0 = f32[1000, 1000]{0,1} transpose(p0), dimensions={1, 0}
  ROOT a0 = f32[1000, 1000] add(p0, transpose_p0)
}
```

`p0` has two output-to-input maps:
- Direct path: `(d0, d1) -> (d0, d1)`
- Via transpose: `(d0, d1) -> (d1, d0)`

This tells XLA that computing one output element may require reading `p0` twice at different locations.

## Indexing map simplification

The default MLIR affine map simplifier cannot leverage bounds information, so it cannot simplify `mod` and `floordiv` expressions effectively. XLA's `IndexingMapSimplifier` uses the known lower and upper bounds of sub-expressions to reduce maps further.

**Example simplifications:**

| Before | Bounds | After |
|--------|--------|-------|
| `(d0, d1) -> (d0 + d1 floordiv 16, d1 mod 16)` | `d0 in [0,6], d1 in [0,14]` | `(d0, d1) -> (d0, d1)` |
| `(d0, d1) -> (-(-11d0 - d1 + 109) floordiv 11 + 9)` | `d0 in [0,9], d1 in [0,10]` | `(d0, d1) -> (d0)` |

Simplification also handles constraints:

- Constraints of the form `lb ≤ affine_expr ≤ ub` are rewritten using tighter bounds derived from variable ranges.
- Constraints that are always satisfied are eliminated entirely.
- Affine expressions inside constraints are simplified using the same rules as the map itself.

A practical consequence: chained reshapes that cancel each other are automatically detected:

```hlo
p0 = f32[10, 10, 10] parameter(0)
reshape1 = f32[50, 20] reshape(p0)
reshape2 = f32[10, 10, 10] reshape(reshape1)
```

After composing and simplifying, the map is `(d0, d1, d2) -> (d0, d1, d2)` — an identity.
